读书笔记 -《Clean Code》- 第2章 有意义的命名

发表于 2022-05-02 分类于 学习

第2章 有意义的命名

2.2 名副其实

“变量、函数或类的名称应该已经答复了所有的大问题。它该告诉你，它为什么会存在，它做什么事，应该怎么用。如果名称需要注释来补充，那就不算是名副其实”
问题：那么注释应该是什么情况下写呢？

2.3 避免误导

“应当避免使用与本意相悖的词。”
“List一词对程序员有特殊意义。如果包纳账号的容器并非真是个List，就会引起错误的判断[3]。所以，用accountGroup或bunchOfAccounts，甚至直接用accounts都会好一些。”
我确实经常用List来命名一组数据。没有在认真上区分它，仅仅是我想到的相近的词语。

2.4 做有意义的区分

“ Variable一词永远不应当出现在变量名中。Table一词永远不应当出现在表名中。”
“要区分名称，就要以读者能鉴别不同之处的方式来区分。”

2.6 使用可搜索的名称

“单字母名称和数字常量有个问题，就是很难在一大篇文字中找出来。”
“窃以为单字母名称仅用于短方法中的本地变量。”

2.8 避免映射思维

“专业程序员善用其能，编写其他人能理解的代码。”

2.9 类名

“类名和对象名应该是名词或名词短语，如Customer、WikiPage、Account和AddressParser。避免使用Manager、Processor、Data或Info这样的类名。类名不应当是动词。”

2.10 方法名

“方法名应当是动词或动词短语，如postPayment、deletePage或save。属性访问器、修改器和断言应该根据其值命名，并依Javabean标准[10]加上get、set和is前缀。”

2.12 每个概念对应一个词

“给每个抽象概念选一个词，并且一以贯之。例如，使用fetch、retrieve和get来给在多个类中的同种方法命名。”

2.16 添加有意义的语境

“设想你有名为firstName、lastName、street、houseNumber、city、state和zipcode的变量。当它们搁一块儿的时候，很明确是构成了一个地址。不过，假使只是在某个方法中看见孤零零一个state变量呢？你会理所当然推断那是某个地址的一部分吗？”

“可以添加前缀addrFirstName、addrLastName、addrState等，以此提供语境。”

思考

读完这一章后，有反思，有困惑。

反思自己之前的代码的命名不规范，不合理。没有做到书中的有意义的命名区分。学着使用文中的规范来约束自己。

困惑的是文中写的很理想化，仅仅使用变量名就阐述了代码本身运行的逻辑；无需多余的注释；然而我在论坛上看到很多人这批判不写注释的人；那么这种不写注释的人如何确保自身的命名能够准确表达，而不被批判呢？

什么是类？- 菜狗理解

发表于 2022-05-02 分类于 学习

类

类(class)在维基百科上的定义：

“在面向对象编程中是一种面向对象电脑编程语言的构造，是创建对象的蓝图，描述了所创建的对象的共同的特性和方法。
类的更严格的定义是有某种特定的元数据所组成的内聚的包。”

提问

Q：那么元数据是什么？

我在《阮一峰的网络日志》里，得到了这样的解释；

元数据（MetaData）
元数据是用来描述数据的数据
Data that describes other data

单单这样说，不太好理解，我来举个例子。

下面是契诃夫的小说《套中人》中的一段，描写一个叫做瓦莲卡的女子：

“（她）年纪已经不轻，三十岁上下，个子高挑，身材匀称，黑黑的眉毛，红红的脸蛋－－一句话，不是姑娘，而是果冻，她那样活跃，吵吵嚷嚷，不停地哼着小俄罗斯的抒情歌曲，高声大笑，动不动就发出一连串响亮的笑声：哈，哈，哈！“

这段话里提供了这样几个信息：年龄（三十岁上下）、身高（个子高挑）、 &nbsp; 相貌（身材匀称，黑黑的眉毛，红红的脸蛋）、性格（活跃，吵吵嚷嚷，不停地哼着小俄罗斯的抒情歌曲，高声大笑）。有了这些信息，我们就可以大致想像出瓦莲卡是个什么样的人。推而广之，只要提供这几类的信息，我们也可以推测出其他人的样子。这个例子中的”年龄”、”身高”、”相貌”、”性格”，就是元数据，因为它们是用来描述具体数据/信息的数据/信息。

我的理解是，元数据就是能够在一定的共识下，能够根据元数据快速了解元数据所描述的东西的一种数据。

类的定义

把类这个字从编程里面给抽到现实世界，那么在什么情况下会使用的类这个字呢。
同类，一类人。
具有相同的特质的就可分为一类。
那么我认为的话，类就是一种共通的东西，既可以用在这个东西；又可以用在那个东西上。
维基百科上的定义如下:

在现实世界中，经常有属于同一个类的对象。例如，某辆自行车只是世界上很多自行车中的一辆。在面向对象软件中，也有很多共享相同特征的不同的对象：矩形、雇用记录、视频剪辑等。可以利用这些对象的相同特征为它们创建一个蓝图。对象的软件蓝图称为类。

类是定义同一类所有对象的变量和方法的蓝图或原型。

类、对象、实例

实例：

它描述了一些对象的行为规则，而这些对象就被称为该类的实例。

类：

读书笔记 -《Clean Code》- 第1章 整洁代码

发表于 2022-05-02 分类于 学习

第1章 整洁代码

1.1 要有代码

“即便是人类，倾其全部的直觉和创造力，也造不出满足客户模糊感觉的成功系统来。”

不知道为什么突然读到这里，想起了《三体》里面的思维透明的三体人和有着想和说不一致的地球人。^-^在我的世界里，人与人之间是永远无法相互理解，只能穷尽一切方法向他人描述自己脑海里面的东西；就如同一个见过瓷器的人，向一个没见过瓷器的人描述瓷器；使用各种形容词，类比等来展现事物的具象。正如同与代码之与需求，需求规约原则在我看来就如同将模糊约束，使其更加具体的东西。

1.2 糟糕的代码

“我们都没听过勒布朗（LeBlanc）法则：稍后等于永不（Later equals never）。”

确实，糟糕的代码就是你明知他有”问题”，但是你在某种原因之下，订下稍后，之后深陷其他沼泽。后来让”问题”解决的成本变得不可估计，最终放弃了解决。所以解决问题最好的时候就是”现在“；还有就是预防问题的产生。

1.3.3 谜题

“赶上期限的唯一方法——做得快的唯一方法 ——就是始终尽可能保持代码整洁。”

按照预定计划，预定规则，坚守纪律完成目标；一定可以得到好的结果。

1.3.4 整洁代码的艺术

“写整洁代码，需要遵循大量的小技巧，贯彻刻苦习得的“整洁感”。”

那么什么是代码感呢?

1.3.5 什么是整洁代码

“糟糕的代码想做太多事，它意图混乱、目的含混。整洁的代码力求集中。每个函数、每个类和每个模块都全神贯注于一事，完全不受四周细节的干扰和污染。”

这句话对我来说非常有用，因为我往往在实现一个功能的时候，总是去思考更多的东西。我的想法是在实现一个功能的时候把他给拆分成几个小部分，然后小部分继续拆分，直至无法拆分。再一个小部分，一个小部分的实现。

“整洁的代码应可由作者之外的开发者阅读和增补。它应当有单元测试和验收测试。它使用有意义的命名。它只提供一种而非多种做一件事的途径。它只有尽量少的依赖关系，而且要明确地定义和提供清晰、尽量少的API。代码应通过其字面表达含义，因为不同的语言导致并非所有必需信息均可通过代码自身清晰表达。”

总结下来，整洁的代码应该具有以下的特点：

  • 具有可读性和拓展性(every developer)；
  • 有单元测试和验收测试；
  • 使用有意义的命名；
  • 尽量少的依赖；
  • 有尽量少的，清晰的API；
  • 代码可以自身阐述自身。

“没有测试的代码不干净。”

书中的简单代码规则：

“能通过所有测试；
没有重复代码；
体现系统中的全部设计理念；
包括尽量少的实体，比如类、方法、函数等。”

一言蔽之:

“减少重复代码，提高表达力，提早构建简单抽象。这就是我写整洁代码的方法。”

思考

对于要有代码，我不可置否；但是目前行业内十分推崇低代码平台，AI写代码等等。其他行业的从业人士开始纷纷唱衰该行业；使得我陷入困惑。但是我认为可能会淘汰掉低端的从业人员，或者说迫使低端从业人员迈向高端吧。
读到糟糕的代码，感觉似乎就是在说我一样，哈哈。
后续有便阐述到，什么是整洁代码，以及从业多年的人员对如何做到整洁代码的心得；让我大获裨益。

读书笔记 -《Clean Code》- 序

序

“对于软件而言，百分之八十或更多的工作量集中在我们美其名曰“维护”的事情上：其实就是修修补补。”

如果在开发的时候，只考虑把功能给实现，那确实会是很简单的。但是如果将“维护”考虑进去，考虑以后软件功能的可拓展性，减少代码之间依赖性等等。开发的细节，原则变得尤为重要。

“良好的软件实践遵循这些规程：专注、镇定和思考。”

专注与思考：最近和小波老师结对交流的时候；小波老师提出issues，我来解决issues，我总是会被issues以外知识诱惑。小波老师告诉我专注于issues来开发，可以使得我更加专注。

“整理（Seiri）[4]，或谓组织（想想英语中的sort（分类、排序）一词）。搞清楚事物之所在——通过恰当地命名之类的手段——至关重要。觉得命名标识无关紧要？读读后面的章节吧。 整顿（Seiton），或谓整齐（想想英文中的systematize（系统化）一词）。有句美国老话说：物皆有其位，而后物尽归其位（A place for everything, and everything in its place）。每段代码都该在你希望它所在的地方——如果不在那里，就需要重构了。
清楚（Seiso），或谓清洁（想想英文中的shine（锃亮）一词）。清理工作地的拉线、油污和边角废料。对于那种四处遗弃的带注释的代码及反映过往或期望的无注释代码，本书作者怎么说的来着？除之而后快。
清洁（Seiketsu），或谓标准化。有关如何保持工作地清洁的组内共识。本书有没有提到在开发组内使用一贯的代码风格和实践手段？这些标准从哪里来？读读看。
身美（Shitsuke）[5]，或谓纪律（自律）。在实践中贯彻规程，并时时体现于个人工作上，而且要乐于改进。”

对于5S标准思考：

整理(Seiri)或者说是sort(分类、排序) :

以对象，属性，函数之类的分类，排序吗？

整顿(Seiton)或者说是整齐(systematize系统化) :

我的理解是分类，归档。

清楚(Seiso)或者说是shine(锃亮) :

冗余，无用的注释要除之而后快。不够整洁，不够清楚，重复逻辑的代码需要整理。

清洁(Seiketsu)或者说是标准化 :遵循标准化，团队内部协作，组内需要约定好代码风格；这样便于团队的开发效率，以及代码的整洁性。个人写代码的时候，也要遵守”规范”，不然一个项目在这个模块是这个风格，另一个模块又是另外一个风格，会造成代码看起来乱七八糟的。

身美(Shitsuke)或者说是纪律(自律) :

生活和工作，贯彻自己订下的规程计划，常常改进。

思考

读完序之后，感受到的是作者向我描述了从事开发这一行业的现状；这种问题是共通的，作者从其他行业的解决问题的经验中，汲取我们这一行业所需要的养分。
